- **Review mode:** 2 per-service agents (auth-service, portal-service) + primary-reviewer inline analysis for the remaining services and all 5 global lenses (agent fleet was cut short by session limits; findings below are evidence-backed from direct diff/file reads, `make test`, and `make sast`).

## Executive summary

This PR is a **near-complete, high-quality adoption of the flywindy/o11y SDK** across the whole platform. All 14 services (plus the 2 data-migration tools) are converted to `pkg/obs.Init`; the bespoke `pkg/otelutil` and the direct `Marz32onE/otel-nats` dependency are gone from first-party code; Mongo/Cassandra/Valkey/Elasticsearch/Vault/NATS/Gin/Resty are all instrumented through consumer-defined option seams; trace continuity across NATS (publish→consume, request→reply, JetStream fetch) is covered by dedicated integration tests. **Estimated o11y integration completeness: ~90%** (per-area breakdown in the Observability chapter).

The dominant risks are operational, not correctness: telemetry-flush ordering is inconsistent in 3 services (worst: search-sync-worker flushes **before** NATS drain), every service now **crashes at startup if `OTEL_SERVICE_NAME` is unset** (a deploy prerequisite living outside this repo), the repo's local compose stack **defines no otel-collector/Tempo/Grafana** despite defaulting every service to `http://otel-collector:4318`, and early-startup fatals regressed from JSON to Go's default text logger in every service that dropped its `slog.SetDefault` bootstrap line.

Reviewability: the code itself is disciplined and consistent, and the 27 commits are well-scoped — but the PR description still says "**planning/docs gate, no production code changes land here**" while the branch ships the entire Phase 0–4 implementation in one PR, abandoning its own locked per-phase rollout plan. The description must be rewritten (or the PR split) before review sign-off; as it stands a reviewer approving "docs only" would be approving 4,900 lines of production change.

## Go expert

- `medium` — **Process-env mutation as API side effect:** `natsutil.enableNATSTracing` (pkg/natsutil/connect.go:78-90) calls `os.Setenv` inside `Connect`. It is well-commented and honors operator overrides, but `os.Setenv` is process-global, not goroutine-safe against concurrent `os.Getenv` elsewhere, and makes `Connect`'s behavior depend on call order. Acceptable stopgap given the SDK exposes no programmatic switch — track upstreaming a `WithTracingEnabled` option in o11y (docs/specs/o11y-followups.md should own this) and remove the env poke.
- `medium` — **Startup logging bootstrap:** every service that deleted `slog.SetDefault(slog.NewJSONHandler…)` now emits pre-`obs.Init` fatals (config parse, obs.Init failure itself) through Go's **text** handler — CLAUDE.md §3 violation on the highest-value log lines. One shared fix: a two-line bootstrap JSON default in `main()` before config parse (or an `init()` in pkg/obs imported for side effect — less magic to keep it explicit in main).
- `low` — `NewHandler(store, collection, bulkSize, tracers ...trace.Tracer)` (search-sync-worker/handler.go): variadic-optional dependency is un-idiomatic; repo style elsewhere is functional options (`mongoutil.Option`) or explicit params.
- `low` — Redundant closure wrappers `func(ctx context.Context) error { return obsShutdown(ctx) }` where `obsShutdown` already has that exact signature (auth-service/main.go:120, portal-service/main.go:141, several workers). Method-value style would be cleaner; consistency across services matters more than the wrapper itself.
- `nitpick` — Gin service-name literals (`"portal-service"`, `"upload-service"`) passed to `o11ygin.Middleware` duplicate `OTEL_SERVICE_NAME`; consider reading from the obs config (expose `sdk`/cfg accessor) so span scope and resource attribute can't drift.
- **Positives worth keeping:** `pkg/obs` is a model thin wrapper — env-standard names, required-vs-default split per CLAUDE.md config rules, providers installed as globals exactly once, doc comment explains *why* (obs.go:1-11). The `Observability` minimal interface duplicated per consumer package (mongoutil/cassutil/valkeyutil/minioutil/searchengine) rather than shared is the *correct* reading of "define interfaces in the consumer" — reviewers should not "deduplicate" it. Error wrapping (`fmt.Errorf("init o11y sdk: %w", err)`) and option-pattern hygiene (`nil`-option guard, obs.go:37) are consistent throughout.
- `low` — `go.mod` still carries `Marz32onE/instrumentation-go/otel-nats v0.2.11 // indirect` — pulled transitively by `flywindy/o11y` itself. Not a leftover; but the PR's "full replacement" claim (D2) should be worded as "no direct dependency" to avoid reviewer confusion.

Recommendation: keep required (it's the right default) but gate the production rollout on manifests being updated first, and say so in the PR body.

## Appendix: Performance deep-dive — does o11y slow the platform down?

Verdict: **no blocking-path hazard exists in the design; the real risk is 100% trace volume (network/collector/storage cost + silent span drops under burst), not latency.** Per-pillar env kill switches (`O11Y_TRACES_ENABLED=false`) make any production regression an env-only rollback. Evidence below is from reading `o11y@v0.8.0` sources in the module cache.

### Mechanism-by-mechanism

1. **Export path is non-blocking (the load-bearing safety property).** The SDK builds its provider with `sdktrace.WithBatcher(exporter)` (o11y `internal/trace/trace.go:61`) — the stock OTel BatchSpanProcessor: queue 2048, batch 512, 5s schedule, **drops spans when the queue is full instead of blocking**, OTLP/HTTP export on a background goroutine. A dead or slow collector cannot stall message processing; the failure mode is silently missing traces.
2. **Hot-path per-message cost is µs-scale; message-gatekeeper has the highest relative share.** Each hop pays: traceparent extract, span start/end (~1–2µs, a few hundred bytes allocated), one histogram record (atomic). Negligible against Cassandra/Mongo round-trips (ms) in the workers — but the gatekeeper only validates + republishes, so its baseline latency is the lowest and instrumentation's *relative* cost the highest. It is also the service that adopted sonic for GC budget; span allocations claw some of that back. This is the one spot worth measuring (see below).
3. **Unset sampler = 100% sampling — first knob to turn.** `configureSampler` (o11y.go:481-494) only installs a ratio when explicitly configured; otherwise OTel defaults to `parentbased_always_on`, and no compose/deploy file in the repo sets `OTEL_TRACES_SAMPLER`. At ~10–20 spans per message platform-wide, 1k msg/s ≈ 15k spans/s/site: the pressure lands on collector/Tempo and network egress, and bursts overflow the 2048-slot queue (silent drops). Fine pre-production; set `parentbased_traceidratio` before scale (action #8 in the main list).
4. **Confirmed non-issues.**
   - *Logging:* `OtelSlogHandler` (internal/log/handler.go:17-35) only injects traceId/spanId from ctx then delegates to the base JSON handler — logs stay on stdout, no OTLP log export. Negligible.
   - *Metrics:* Prometheus pull-based; hot path pays one histogram record per op; labels are fixed semconv sets (redis hook pre-builds `poolMetricAttrs`, hook.go:94), gin uses route templates — **no cardinality-explosion path**.
   - *Valkey:* every worker call site sets `WithRequireParentSpan(true)`; unparented background probes short-circuit before span creation (redis/hook.go:339) — the PR already made the right cost/noise call here.
   - *DB spans:* one span + one record per operation is noise-level against ms-scale I/O.
5. **Known pure waste** (already in the findings): unfiltered `/healthz`/`/readyz` probe spans on the 3 Gin services; OTLP retry + error logs when no collector exists (docker-local today); minor goroutine churn — `wrapMessageBatch` spawns one goroutine per `Fetch` (o11y nats/jetstream.go:473), negligible at search-sync-worker fetch rates.

### Recommended verification & guardrails

1. **A/B the gatekeeper with the repo's own loadgen** (`tools/loadgen`, `maxrps_messages`): identical run with `O11Y_TRACES_ENABLED=false` vs on; compare max RPS and p99. Pillar toggles make this a pure env A/B. Suggested as a Phase-4 acceptance item.
2. **Set a sampling ratio in deploy env before production load** — no code change required.
3. **Dashboard the BatchSpanProcessor dropped-span counter** — queue overflow is otherwise invisible; the system will look healthy while traces quietly thin out.
4. **Document the kill switch** in the runbook: `O11Y_TRACES_ENABLED=false` (and siblings) is a zero-deploy rollback for any telemetry-suspected regression.
