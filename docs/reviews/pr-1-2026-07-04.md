## Executive summary

This PR is a **near-complete, high-quality adoption of the flywindy/o11y SDK** across the whole platform. All 14 services (plus the 2 data-migration tools) are converted to `pkg/obs.Init`; the bespoke `pkg/otelutil` and the direct `Marz32onE/otel-nats` dependency are gone from first-party code; Mongo/Cassandra/Valkey/Elasticsearch/Vault/NATS/Gin/Resty are all instrumented through consumer-defined option seams; trace continuity across NATS (publish→consume, request→reply, JetStream fetch) is covered by dedicated integration tests. **Estimated o11y integration completeness: ~90%** (per-area breakdown in the Observability chapter).

The dominant risks are operational, not correctness: telemetry-flush ordering is inconsistent in 3 services (worst: search-sync-worker flushes **before** NATS drain), every service now **crashes at startup if `OTEL_SERVICE_NAME` is unset** (a deploy prerequisite living outside this repo), the repo's local compose stack **defines no otel-collector/Tempo/Grafana** despite defaulting every service to `http://otel-collector:4318`, and early-startup fatals regressed from JSON to Go's default text logger in every service that dropped its `slog.SetDefault` bootstrap line.

Reviewability: the code itself is disciplined and consistent, and the 27 commits are well-scoped — but the PR description still says "**planning/docs gate, no production code changes land here**" while the branch ships the entire Phase 0–4 implementation in one PR, abandoning its own locked per-phase rollout plan. The description must be rewritten (or the PR split) before review sign-off; as it stands a reviewer approving "docs only" would be approving 4,900 lines of production change.

## Service: auth-service

Diff surface: `main.go` (obs.Init, gin middleware reorder, shutdown hook), `deploy/docker-compose.yml` (OTEL env). No handler/route/store changes.

**(a) Diff correctness**
- `medium` — Removing `slog.SetDefault(slog.NewJSONHandler…)` from `main()` (main.go:36-41) means any failure **before** `obs.Init` succeeds (config parse main.go:44-47, bad signing key :49-52, obs.Init itself :56-59) logs `slog.Error("fatal error")` through Go's default **text** handler — violates CLAUDE.md "always JSON" on exactly the startup-failure lines ops greps for. Same regression in portal/upload and the NATS services.
- `low` — On early error returns after `obs.Init` (e.g. OIDC validator failure main.go:79-81), `obsShutdown` is never called; buffered spans lost on startup failure.
- `nitpick` — `func(ctx context.Context) error { return obsShutdown(ctx) }` (main.go:120) could be `obsShutdown` directly.
- Middleware order is correct per the SDK's canonical chain (o11y before `gin.Recovery()`); CORS-before-tracing is deliberate and commented (main.go:88).

**(b) Scope drift** — none; minimal, on-purpose diff.

**(c) Abstraction changes** — justified; consumes shared `pkg/obs.Init` instead of inlining SDK setup; deleted bespoke `pkg/otelutil` correctly superseded.

**(d) Design coherence**
- Fits; identical wiring shape to portal-service/upload-service.
- `medium` — new hard dependency: `obs.Config.ServiceName` is `env:"OTEL_SERVICE_NAME,required"` (pkg/obs/obs.go:30) → auth-service **crashes at startup** if unset. Compose sets it, but production manifests live outside this repo — a rollout prerequisite that must be called out in the PR.

**(e) Project-pattern adherence** — `shutdown.Wait` retained, correct order (server drain → obs flush last). `low` (pre-existing, in-scope for an o11y PR): handler.go:172, :221 use `slog.Debug` without `Context` — no traceId even with the new SDK logger.

**(f) Client-API doc rule** — no violation; no auth-service route/schema changed. The shared CORS change (pkg/ginutil/middleware.go:37 adds `traceparent/tracestate/baggage`) is wire-visible but not a schema change.

**(g) o11y status — ~80%.** Done: obs.Init, gin server spans+metrics, propagator, SDK slog default, graceful shutdown, CORS trace headers, compose envs. Gaps: `medium` no `o11ygin.WithSkipPaths()` → `/healthz`+`/readyz` traced on every kube probe (SDK ships `DefaultSkipPaths`; fix uniformly with portal/upload); `medium` access log not trace-correlated (pkg/ginutil/middleware.go:53 uses `slog.Info` without ctx); `medium` `pkg/oidc.Validator` uses a bare `net/http.Client` (pkg/oidc/oidc.go:53,75) — JWKS/token calls produce no client spans (also a pre-existing "use Resty" violation); `low` request_id not attached as span attribute (trace↔request-id pivot is one-directional).
